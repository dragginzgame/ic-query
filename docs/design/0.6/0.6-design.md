# ic-query 0.6 Design: Public Source Adapters

## Status

- Status: active implementation note
- Release line: `0.6.x`
- Decision style: public source-adapter API
- Primary implementation target: downstream crates that need typed reports
  without always using the built-in live host adapters

## Summary

0.6 starts the public source-adapter line. The 0.5 line opened request,
report, cache, refresh, and rendering APIs, but source traits stayed internal.
The next step is to expose custom source boundaries incrementally, beginning
with the generic ICRC family because it is live-only and already has
fixture-backed report construction behind an internal trait.

## Goals

1. Let host-only downstream crates provide custom source implementations for
   report builders without enabling `cli`.
2. Keep source APIs report-oriented and typed, not raw CLI argument or process
   boundaries.
3. Preserve explicit source endpoints in request/report DTOs so live network
   use remains visible.
4. Open source families one at a time, with public smoke tests for each family.

## 0.6.0 Slice

The first 0.6.0 slices expose generic ICRC, subnet catalog, and NNS registry
custom sources under `features = ["host"]`.

Generic ICRC:

- `IcrcSource`
- `LiveIcrcSource`
- source data DTOs such as `IcrcTokenData` and `IcrcTransactionsData`
- `build_icrc_*_report_with_source` builders

Subnet catalog:

- `SubnetCatalogSource`
- `SubnetCatalogSourceRequest`
- `LiveNnsRegistryRefreshSource`
- `load_or_refresh_subnet_catalog_with_source`
- `refresh_subnet_catalog_with_source`
- `build_subnet_catalog_list_report_with_source`
- `build_subnet_catalog_info_report_with_source`

NNS registry:

- `NnsRegistrySource`
- `NnsRegistrySourceRequest`
- `NnsRegistryVersionData`
- `LiveNnsRegistrySource`
- `build_nns_registry_version_report_with_source`

These keep the existing built-in report functions intact while allowing
downstream host tools to inject fixture, cached, proxy, or alternate transport
sources.

## 0.6.1 Slice

The 0.6.1 slice opens NNS inventory source adapters under `features =
["host"]`.

NNS inventory:

- `NnsNodeSource`
- `NnsDataCenterSource`
- `NnsNodeProviderSource`
- `NnsNodeOperatorSource`
- matching `*SourceRequest` DTOs
- matching `LiveNns*Source` adapters
- matching list/info `*_with_source` builders
- matching refresh `*_with_source` wrappers

These adapters use public list reports as their source return contracts rather
than exposing the lower-level registry inventory structs. That keeps registry
wire/projection details private while still letting downstream host tools
inject fixture, cached, proxy, or alternate transport inventory sources.

## Deferred Work

NNS proposal/topology and SNS source traits need separate passes because
several internal traits still expose governance, topology-join, or SNS-specific
source DTOs that were not designed as public contracts. Those traits should be
opened only after their request/data boundaries are reviewed family by family.
