# ic-query 0.4.0 Design: Library Boundary And CLI Split

## Status

- Status: active planning note
- Release line: `0.4.0`
- Decision style: crate boundary, runtime compatibility, and downstream API
- Primary implementation target: reusable Rust library plus the existing `icq`
  CLI as a wrapper
- Current state: the package already has an implicit library target, but the
  useful API is mostly CLI-oriented and many report/query modules remain behind
  private crate roots

## 1. Summary

`ic-query` should become usable by two downstream Rust consumers:

1. Native host tools such as Canic or another CLI.
2. Internet Computer canisters that want the same request/report logic without
   host-only dependencies.

0.4.0 should split the project by runtime boundary:

```text
ic-query        reusable library: models, requests, reports, source traits,
                pure sorting/filtering/projection, optional host adapters

ic-query-cli    CLI package: clap parsing, process args, stdout/stderr,
                command help, progress output, and the `icq` binary
```

Report builders should depend only on source traits and report requests. They
should not depend on the transport, cache layer, process environment, or
terminal behavior used to obtain the data. CLI parsing and process IO should
live only in the CLI package.

## 2. Problem

Before the package split, the code exposed `run_from_env()` and `run()` from
the library, with the binary acting as a thin wrapper. That was useful for
testing the CLI, but it was not a clean downstream API:

- `run()` takes process-style arguments instead of typed requests.
- `clap`, usage strings, stdout, stderr, and exit behavior are near the public
  boundary.
- Native host behavior such as project-root lookup, cache files, refresh locks,
  progress lines, and `ic-agent` calls is mixed with reusable report logic.
- Canister consumers cannot depend on filesystem, process, Tokio runtime, or
  `ic-agent` behavior.

The split should turn existing report construction into a stable, typed API
without forcing downstream consumers to shell out to `icq` or reimplement IC
metadata conversions.

## 3. Goals

1. Provide a typed Rust API for NNS, SNS, ICRC, and subnet-catalog report
   construction.
2. Keep the `icq` binary behavior and command output stable unless a change is
   explicitly accepted for 0.4.0.
3. Let native CLI consumers use host adapters, cache helpers, and text/JSON
   renderers without depending on `clap`.
4. Let canister consumers compile the core library without `ic-agent`, Tokio,
   filesystem cache code, project-root lookup, progress rendering, or process
   IO.
5. Keep live host calls behind source traits so unit tests and canister
   adapters can provide fixture or inter-canister sources.
6. Preserve the cache invariant that cache keys describe collected data, not
   rendered views.
7. Keep JSON report fields raw and script-friendly; keep text rendering
   human-facing and optional for downstream consumers.

## 4. Non-Goals

0.4.0 should not:

1. Add mutating IC operations.
2. Force downstream tools to use the `icq` CLI argument model.
3. Require canister consumers to use local cache files.
4. Stabilize every internal helper as public API in one release.
5. Add a canister storage backend for complete snapshots.
6. Hide live network or inter-canister calls behind APIs that look cache-only.

## 5. Package Layout

The intended workspace shape is:

```text
Cargo.toml
crates/
  ic-query/
    Cargo.toml
    src/lib.rs
    tests/...
  ic-query-cli/
    Cargo.toml
    src/main.rs
    src/cli/...
```

The repository root is a virtual workspace. Both publishable package roots live
under `crates/`, with `crates/ic-query` owning the reusable library and
`crates/ic-query-cli` owning the binary package.

`ic-query` owns reusable domain code:

```text
ic_query::nns
ic_query::sns
ic_query::icrc
ic_query::subnet_catalog
ic_query::source
ic_query::text
ic_query::json       feature = "json"
ic_query::cache      feature = "cache"
ic_query::host       feature = "host"
```

`ic-query-cli` owns process-facing code:

```text
clap command definitions
argument parsing
usage/help/version output
stdout/stderr writers
terminal progress lines
process exit mapping
project-root lookup defaults
```

The `ic-query-cli` package should continue to publish a binary named `icq`.

## 6. Crates.io Naming Decision

0.4.0 should repurpose the existing `ic-query` package name for the library and
publish the CLI wrapper as `ic-query-cli`:

```text
ic-query      library package, imported as `ic_query`
ic-query-cli  CLI package, installs the `icq` binary
```

This gives the reusable library the natural package name. It changes the
install command from:

```bash
cargo install ic-query
```

to:

```bash
cargo install ic-query-cli
```

That install-command change is acceptable for 0.4.0 because the project has no
known downstream package users to preserve. The release notes must still call
out the change explicitly so existing local users know how to install the
binary.

A compatibility-preserving alternative would be:

```text
ic-query-core  library package
ic-query       CLI package, installs `icq`
```

0.4.0 should not take that path unless new compatibility constraints appear
before implementation starts.

## 7. Feature Model

The library should be lean by default so canister builds and native consumers
do not accidentally pull host-only dependencies:

```toml
[features]
default = []
std = []
serde = ["dep:serde"]
json = ["serde", "dep:serde_json"]
text = ["std"]
host = ["std", "dep:ic-agent"]
cache = ["std", "json"]
cache-refresh = ["cache", "host"]
```

The exact dependency list can be refined during implementation. The important
boundary is:

- `serde` derives serialization and deserialization for report and snapshot
  models.
- `json` provides `serde_json` helpers for loading, storing, and rendering JSON.
- `text` provides human-facing pure renderers.
- `host` may depend on `ic-agent`, native TLS choices, endpoint clients, and
  host clock helpers.
- `cache` may depend on filesystem paths, atomic writes, refresh locks, and
  JSON snapshot files, but not live host-call adapters.
- `cache-refresh` combines local cache storage with host live-call behavior for
  native refresh workflows.
- A dedicated `canister` feature should not exist in 0.4.0 unless it enables a
  concrete adapter implementation. The canister-compatible core should work
  with `--no-default-features`.

0.4.0 promises canister compatibility, not true `no_std` compatibility. The
core `ic-query --no-default-features` library must compile for
`wasm32-unknown-unknown` without host-only dependencies. It may still use
`std`, `String`, `Vec`, formatting, `serde`, and ordinary Rust error types when
those are available to IC canister crates.

True `no_std` or `alloc`-only support is out of scope unless explicitly added
to this design. If that goal is added later, the design must account for
allocator setup, owned strings and vectors, formatting, serde support, and
public error types under `no_std`.

For 0.4.0, canister support means the core traits and models compile for
`wasm32-unknown-unknown` with no host dependencies. Concrete `ic-cdk` adapters
are out of scope unless explicitly added. A future adapter can live in a
separate crate such as `ic-query-ic-cdk`, behind an `ic-cdk` feature, or as an
example canister source.

## 8. Public API Shape

The core library should expose typed request and report models. A native tool
or canister should call report builders directly instead of synthesizing CLI
arguments.

Requests describe what report to build. Sources describe where data comes
from. Reports carry the final request summary, source descriptor, collection
timestamp or registry version where available, and report rows.

Public request APIs should avoid raw strings for IC identifiers. Prefer
existing typed identifiers or thin wrappers:

```text
candid::Principal
NetworkId
CanisterId
LedgerCanisterId
SnsRootCanisterId
SubnetId
```

Request constructors or builders should validate invariants before report
builders run. Examples include nonzero limits, valid sort/filter combinations,
and command-family-specific bounds.

Example native CLI consumer:

```rust
use ic_query::{NetworkId, ReportError};
use ic_query::nns::topology::{
    NnsTopologySource, NnsTopologySummaryRequest,
    build_nns_topology_summary_report,
};

async fn load_summary<S>(source: &S) -> Result<(), ReportError<S::Error>>
where
    S: NnsTopologySource,
{
    let request = NnsTopologySummaryRequest {
        network: NetworkId::mainnet(),
    };
    let report = build_nns_topology_summary_report(request, &source).await?;
    println!("{}", ic_query::text::nns_topology_summary_report(&report));
    Ok(())
}
```

Example canister-style consumer:

```rust
use candid::Principal;
use ic_query::sns::proposals::{
    ListProposalsCall, ListProposalsResponse, SnsGovernanceSource,
    SnsProposalsRequest, build_sns_proposals_report,
};
use std::num::NonZeroU32;

struct CanisterSnsSource;

impl SnsGovernanceSource for CanisterSnsSource {
    type Error = String;

    fn source_descriptor(&self) -> ic_query::SourceDescriptor {
        ic_query::SourceDescriptor::canister_call("sns_governance")
    }

    async fn list_proposals(
        &self,
        request: ListProposalsCall,
    ) -> Result<ListProposalsResponse, Self::Error> {
        // Implement with inter-canister calls in the consuming canister.
    }
}

async fn proposals(root: Principal) -> Result<(), String> {
    let request = SnsProposalsRequest::builder(root)
        .limit(NonZeroU32::new(25).expect("nonzero limit"))
        .build()
        .map_err(|err| err.to_string())?;
    let _report = build_sns_proposals_report(request, &CanisterSnsSource).await?;
    Ok(())
}
```

The examples are illustrative. Implementation should reuse existing request,
report, source, and conversion types where possible rather than inventing
parallel APIs.

The library should expose structured report-builder errors rather than making
`anyhow` part of the public contract:

```rust
pub enum ReportError<E> {
    Source(E),
    InvalidRequest(InvalidRequestError),
    Decode(DecodeError),
    Cache(CacheError),
    InvariantViolation(String),
}
```

Report builders should return:

```rust
Result<Report, ReportError<S::Error>>
```

Downstream CLIs can convert these errors into `anyhow` or process text.
Canister consumers can convert them into compact strings, Candid variants, or
reject-code-specific errors.

## 9. Source Traits

Report builders should depend on source traits that describe the data needed by
that report family:

```text
RegistrySource
NnsGovernanceSource
SnsWasmSource
SnsGovernanceSource
IcrcLedgerSource
```

Trait methods should represent source operations, not rendered commands:

```text
registry_version
subnet_records
node_records
list_deployed_snses
get_sns_metadata
list_sns_proposals
get_sns_proposal
list_sns_neurons
icrc1_metadata
icrc1_balance_of
icrc2_allowance
icrc3_get_blocks
```

The host implementation can wrap `ic-agent`. Canister consumers can implement
the same traits with inter-canister calls. Tests can implement them with
fixtures.

0.4.0 should start with source traits that use `async fn` and static dispatch:

```rust
pub trait SnsGovernanceSource {
    type Error;

    fn source_descriptor(&self) -> SourceDescriptor;

    async fn list_proposals(
        &self,
        request: ListProposalsCall,
    ) -> Result<ListProposalsResponse, Self::Error>;
}
```

Source traits are not guaranteed to be `dyn` compatible in 0.4.0. Report
builders should be generic over source traits rather than accepting trait
objects:

```rust
pub async fn build_sns_proposals_report<S>(
    request: SnsProposalsRequest,
    source: &S,
) -> Result<SnsProposalsReport, ReportError<S::Error>>
where
    S: SnsGovernanceSource,
{
    // ...
}
```

Core source-trait futures should not require `Send`. IC canister
implementations may naturally produce non-`Send` futures. Native host adapters
may add separate `Send` bounds in host-only helper APIs if needed.

The same public traits should support host adapters, canister adapters, and
fixtures. Parallel host-only and canister-only traits should be avoided unless
the shared trait becomes unworkable.

Source identity should live on the source, not on request structs. Requests
select the subject and view semantics. Sources return a `SourceDescriptor`.
Reports include that descriptor so provenance cannot diverge between
a request-declared endpoint and the actual adapter endpoint.

For host calls the descriptor is usually the URL endpoint. For canister calls
it may be a canister principal plus method name or another explicit source
descriptor.

## 10. Cache Boundary

Cache storage and inspection belong behind the `cache` feature. Live cache
refresh belongs behind `cache-refresh`, because a downstream crate may want to
read or inspect local snapshots without pulling `ic-agent` or host live-call
adapters.

Keep the existing cache policy:

```text
Snapshot key = collected data identity
View options = sort, limit, filters, verbosity, format
```

A versioned snapshot key should describe the logical collected data:

```rust
pub struct SnapshotKey {
    pub schema_version: u32,
    pub family: ReportFamily,
    pub network: Option<NetworkId>,
    pub source: SourceDescriptor,
    pub collected_subject: CollectedSubject,
}
```

The implementation should distinguish these independently versioned concepts:

```text
Snapshot schema version
Report schema version
Cache layout version
Rendered text format
```

Those evolve at different rates and should not be collapsed into one version
number.

Cache APIs should accept explicit roots and logical keys:

```text
cache::load_snapshot::<NnsTopologySnapshot>(&root, &key)
cache::store_snapshot(&root, &key, &snapshot, policy)
cache::inspect_snapshot_status(&root, &key)
```

Family-specific report layers should own how complete snapshots are collected.
Avoid committing to one overly generic public refresh function until multiple
families prove they fit the same abstraction. Shared lower-level cache storage
is fine; collection and refresh orchestration can remain family-specific where
that keeps invariants clearer.

The CLI can choose project-local defaults such as `.icq/`, but the library
should not assume the current process is running inside a repository.

## 11. Text And JSON Rendering

Report models should be serializable behind the `serde` feature. JSON helpers
should live behind `json`. JSON output should remain raw and script-friendly.

Text rendering can live in the library if it is pure:

```text
report model -> String
```

Process output should live in the CLI:

```text
String -> stdout/stderr/output file
```

This lets another CLI reuse the same text tables while still owning its own
command model.

Test expectations:

- JSON golden tests must prove report schemas do not change during the crate
  split.
- Text snapshot tests must prove existing CLI output does not change
  accidentally.

## 12. Migration Plan

1. Add golden tests around current CLI JSON, text, and cache behavior where
   coverage is missing.
2. Create the workspace and move package roots under `crates/` mechanically,
   preserving behavior.
3. Move process-facing CLI helpers into `ic-query-cli`.
4. Promote one report family into a public typed API.
5. Introduce or formalize source traits for that report family.
6. Make `ic-query-cli` call the typed API for that family.
7. Repeat the API promotion family by family.
8. Move host `ic-agent` implementations behind the `host` feature.
9. Move filesystem snapshot/cache helpers behind the `cache` feature and live
   refresh helpers behind `cache-refresh`.
10. Keep text rendering pure and callable from both the CLI and native
    downstream tools.
11. Add canister-compatibility checks for the core library.
12. Update README, detailed changelog, and root changelog for any new install
    command or public API surface.

The migration should separate mechanical file moves from behavior or API
changes so each step remains reviewable. Avoid broad module renames unless a
file is being moved to the new crate boundary.

## 13. Compatibility And SemVer

`0.4.0` may make breaking changes freely when they make the library boundary,
CLI wrapper, cache model, or downstream API cleaner. The project is pre-1.0 and
has no compatibility burden that should block the split.

Compatibility expectations are intentionally light:

- Existing behavior may change if the new crate boundary or typed API warrants
  it.
- CLI commands, install commands, JSON report schemas, text output, and cache
  paths may change in 0.4.0.
- Breaking changes should be intentional, documented, and covered by migration
  notes where a user or downstream crate would need to act.
- Public library APIs introduced in 0.4.0 should be documented as the supported
  downstream surface.
- Internal helpers should stay private until they are intentionally stable.

## 14. Testing And Validation

0.4.0 validation should include:

```text
cargo fmt --all -- --check
cargo clippy --workspace --all-targets --all-features
cargo test --workspace --all-features
cargo check -p ic-query --no-default-features
cargo check -p ic-query --target wasm32-unknown-unknown --no-default-features
cargo tree -p ic-query --no-default-features -e features
cargo tree -p ic-query --target wasm32-unknown-unknown --no-default-features -e features
cargo check -p ic-query-cli
git diff --check
```

If the wasm target is not installed locally, CI should install it or run an
equivalent canister-compatibility check.

Feature-specific checks should verify that:

- `ic-query --no-default-features` does not compile host-only code.
- `ic-query --features host` compiles native live source adapters.
- `ic-query --features cache` compiles cache storage and inspection helpers.
- `ic-query --features cache-refresh` compiles host-backed cache refresh
  helpers.
- `ic-query-cli` compiles and runs the existing command test suite.

CI or a local guard should make these negative dependency assertions for the
core library:

- `--no-default-features` must not include `clap`.
- `--no-default-features` must not include `ic-agent`.
- `--no-default-features` must not include Tokio runtime setup.
- `--no-default-features` must not include filesystem cache modules.

If the project adopts `cargo-hack`, add:

```text
cargo hack check -p ic-query --feature-powerset
```

## 15. Open Decisions

1. Whether true `no_std` or `alloc`-only support is a future goal. 0.4.0 only
   promises `wasm32-unknown-unknown` compatibility without host dependencies.
2. Whether source traits stay `async fn` plus static dispatch only, or also get
   dyn-compatible wrappers with boxed futures.
3. Whether host convenience APIs should provide `Send` future variants while
   keeping core source traits non-`Send`.
4. The exact structured `ReportError<E>` and cache error variants exposed by
   public report builders.
5. Whether text renderers are part of the stable public API or exposed as a
   best-effort helper module.
6. Whether public model serialization uses an optional `serde` feature exactly
   as described, or whether core report models always derive serde traits.
7. Which cache APIs are public in 0.4.0: inspection and load only, or also
   family-specific refresh helpers under `cache-refresh`.
8. The concrete `SourceDescriptor` shape and how much source provenance appears
   in report metadata.
9. What MSRV the public async/source-trait API requires.
10. Whether a concrete `ic-cdk` adapter is deferred to a separate crate,
    feature-gated module, or example.

## 16. Risks

- Moving code before the public API is clear could turn internal structure into
  accidental downstream contract.
- A library-only `ic-query` package would surprise existing users of
  `cargo install ic-query`.
- Canister compatibility can regress easily if host dependencies remain in
  default features.
- Source traits that are too broad will be hard for canisters and tests to
  implement; source traits that are too narrow may duplicate fetch logic across
  report families.
- Public cache APIs can accidentally expose path layout as contract instead of
  logical cache keys.

## 17. Initial Success Criteria

0.4.0 is successful when:

1. `ic-query-cli` builds and installs the `icq` binary.
2. Another native crate can depend on `ic-query` and build at least one NNS,
   one SNS, and one ICRC report without invoking CLI argument parsing.
3. `ic-query --no-default-features` compiles without `ic-agent`, `clap`,
   filesystem cache code, or Tokio runtime setup.
4. The existing CLI test suite passes through the new wrapper.
5. Existing JSON schemas and cache paths remain stable unless explicitly
   changed and documented.
